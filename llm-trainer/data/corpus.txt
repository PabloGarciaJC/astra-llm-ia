Pablo conoce los principios SOLID para desarrollar software orientado a objetos de forma estructurada. El principio de responsabilidad única indica que cada clase debe tener una única responsabilidad. El principio abierto cerrado establece que las clases deben permitir extensiones sin modificar el código existente. El principio de sustitución de Liskov garantiza que los objetos derivados pueden reemplazar a los de la clase base. El principio de segregación de interfaces evita que las clases implementen métodos innecesarios. El principio de inversión de dependencias indica que las clases deben depender de abstracciones y no de implementaciones concretas.

Pablo conoce los microservicios como arquitectura que divide una aplicación en múltiples servicios pequeños, independientes y autónomos. Cada servicio maneja una funcionalidad específica, puede desarrollarse de forma independiente y se comunica mediante APIs. Las ventajas incluyen escalabilidad real, mayor resiliencia, desarrollo más veloz y mantenimiento simplificado. Los desafíos incluyen mayor complejidad operativa y necesidad de monitoreo avanzado.

Pablo tiene conocimientos sobre servidores web y ha escrito sobre Apache y Nginx. Apache fue lanzado en 1995 y destaca por su flexibilidad y módulos dinámicos. Nginx fue creado en 2004 con arquitectura asíncrona basada en eventos, ideal para alto tráfico y contenido estático. Muchas arquitecturas modernas combinan ambos usando Nginx como proxy reverso y Apache como servidor de aplicaciones.

Pablo ha escrito sobre el caché web, explicando que es un almacenamiento temporal de datos que permite a sistemas y navegadores acceder rápidamente a información ya solicitada previamente. El caché acelera la carga de páginas, reduce el consumo de ancho de banda y mejora la experiencia del usuario. Existen tres tipos principales: caché del navegador, caché del servidor y caché de aplicaciones. Entre las buenas prácticas destacan definir la caducidad de los recursos, controlar versiones de archivos y combinar el caché con optimización de imágenes y minificación.

Pablo también ha escrito sobre XML, un formato diseñado para almacenar y transportar datos de manera estructurada y legible tanto por humanos como por máquinas. Los archivos XML contienen etiquetas, atributos y valores. Sus usos comunes incluyen el intercambio de datos entre servicios web y APIs, almacenamiento de configuraciones, documentación estructurada y sitemaps para motores de búsqueda. Como desventaja, es más pesado que JSON y su procesamiento es más lento con archivos grandes.

Pablo ha explicado la diferencia entre librerías y frameworks. Una librería es un conjunto de funciones predefinidas donde el control lo tiene el desarrollador, como jQuery, Lodash o Axios. Un framework proporciona una estructura completa donde el control lo tiene el propio framework, como Angular, Vue.js o React. La diferencia principal radica en quién controla el flujo del programa. Los frameworks son ideales para proyectos grandes y escalables, mientras que las librerías ofrecen mayor flexibilidad.

Pablo ha escrito sobre despliegues web, explicando que es el conjunto de pasos para poner en funcionamiento una aplicación en un entorno accesible públicamente. Un despliegue bien estructurado minimiza errores en producción y acelera la entrega de nuevas funcionalidades. Las fases típicas incluyen versionado con Git, construcción, pruebas automáticas mediante pipelines CI/CD, despliegue en servidores y monitoreo continuo. Pablo conoce las promesas en JavaScript, objetos que representan la eventual finalización o fallo de una operación asíncrona. Las promesas surgieron para solucionar el problema del callback hell. Tienen tres estados: pendiente, cumplida y rechazada. Permiten encadenar operaciones asíncronas de forma secuencial y mantienen un flujo lógico similar al código síncrono. Pablo aplica buenas prácticas como manejar errores explícitamente, evitar anidar promesas innecesariamente y aplicar principios SOLID.

Pablo también ha explicado los callbacks, funciones que se transmiten como parámetro a otra función y se ejecutan cuando finaliza una operación específica. Son fundamentales en programación asíncrona, permitiendo que las aplicaciones continúen ejecutándose mientras tareas se resuelven en segundo plano. El problema principal de los callbacks es el callback hell cuando se anidan múltiples niveles, lo que impulsó soluciones modernas como promesas y async/await. Pablo recomienda evitar el anidamiento excesivo, manejar errores explícitamente y usar nombres descriptivos.
